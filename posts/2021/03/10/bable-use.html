babel本身只关系语法，不关心api。 babel不会处理模块化的内容（webpack处理模块化）

babel与webpack区别

  • babel --> JS新语法的编译工具，不关心模块化
  • webpack --> 打包构建工具，是多个loader、plugin的集合，处理模块化

# babel环境搭建

package.json添加内容

  "devDependencies": {
    "@babel/cli": "^7.7.5",
    "@babel/core": "^7.7.5",
    "@babel/plugin-transform-runtime": "^7.7.5", // 可以稍后安
    "@babel/preset-env": "^7.7.5"
  },
  "dependencies": {
    "@babel/polyfill": "^7.7.0", // 可以稍后安
    "@babel/runtime": "^7.7.5" // 可以稍后安
  }

babelrc文件配置

{
    "presets": [
        [
            "@babel/preset-env",
        ]
    ],
    "plugins": [
    ]
}

在terminal中执行下面命令，会将src/index.js中的es6编译成es5。在用webpack时用babel-loader将es6转成es5

npx babel src/index.js

preset-env是一堆plugin的集合，这样我们就不用一个个引入plugin了。如果不能满足我们的需求，我们可以在plugins处扩展引入

# .babelrc基本配置

babel-polyfill按需引入

{
    "presets": [
        [
          // 这样配置是corejs的按需引入
            "@babel/preset-env",
            {
                "useBuiltIns": "usage", // 按需引入。就不需要写在文件中手动全部引入了。
                "corejs": 3 // 现在版本是3
            }
        ]
    ],
    "plugins": [
        [
            "@babel/plugin-transform-runtime", // 解决全局变量污染问题babel-runtime
            {
                "absoluteRuntime": false,
                "corejs": 3,
                "helpers": true,
                "regenerator": true,
                "useESModules": false
            }
        ]
    ]
}

# babel-polyfill

core-js并不支持generator语法，需使用时要单独引入。而babel-polyfill是两都的集合。

  • babel7.4之后弃用babel-polyfill 推荐直接使用core-js和regenerator

遇到问题：

  • 会污染全局环境
  • 如果做一个独立的web系统则无碍
  • 如果做一个第三方lib，则会有问题

解决方案：

  • 通过babel-runtime解决污染全局变量问题
  • 产出第三方lib要用babel-runtime

# 为什么proxy不能被polyfill

  • class可以通过funtion模拟出来
  • promise可以通过回调模拟出来
  • proxy 功能通过object.defineProperty无法模拟 glitzma/glitzma.github.iohttps://github.com/glitzma/glitzma.github.io.git
